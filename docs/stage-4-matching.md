# Етап 4 — Deterministic Matching

## Мета

Етап 4 перетворює загальну стрічку оголошень на персональну видачу для конкретного `SearchProfile`. Оцінка не використовує LLM і не залежить від випадковості: однаковий профіль та однакове оголошення завжди дають однаковий результат.

## Формула

Підсумковий Match Score — зважене середнє шести компонентів:

| Компонент | Вага |
|---|---:|
| Бюджет | 30% |
| Локація | 25% |
| Кількість кімнат | 15% |
| Тип житла | 10% |
| Особливі умови | 10% |
| Повнота даних | 10% |

Кожний компонент повертає:

- оцінку від 0 до 100;
- статус `strong`, `partial`, `miss` або `unknown`;
- коротке пояснення українською;
- стабільний код компонента;
- вагу у фінальному результаті.

## Hard mismatch

Варіант вважається `eligible=false`, якщо:

- місто не відповідає профілю;
- район явно виключений;
- перевищення бюджету настільки велике, що бюджетний компонент дорівнює нулю.

За замовчуванням endpoint приховує такі варіанти, але контракт підтримує `eligible_only=false` для майбутніх адміністративних та аналітичних сценаріїв.

## API

```text
GET /api/v1/search-profiles/{profile_id}/matches/
```

Параметри:

- `min_score` — 0–100;
- `eligible_only` — `true` або `false`;
- `ordering` — `-match_score`, `match_score`, `-published_at`, `price_uah`;
- `limit` — 1–500.

Приклад відповіді:

```json
{
  "profile": {"id": "...", "name": "Львів до 22к", "city": "Львів"},
  "count": 12,
  "results": [
    {
      "listing": {"id": "...", "title": "1-кімнатна квартира", "price_uah": 19000},
      "match": {
        "score": 88,
        "eligible": true,
        "summary": "Майже повна відповідність вашому пошуку.",
        "strengths": ["Ціна входить у заданий діапазон."],
        "compromises": [],
        "unknowns": [],
        "components": []
      }
    }
  ],
  "meta": {"algorithm": "deterministic-v1"}
}
```

## Безпека

- профіль отримується тільки через user-scoped queryset;
- чужий `profile_id` повертає `404`;
- у matching потрапляють лише активні оголошення з увімкнених та юридично схвалених джерел;
- query parameters проходять DRF-валідацію;
- endpoint не приймає довільних ваг від клієнта;
- алгоритм не робить зовнішніх запитів і не виконує користувацький код.

## Mini App

Стрічка етапу 4 дозволяє:

- вибрати активний пошуковий профіль;
- задати мінімальний Match Score;
- сортувати за відповідністю, ціною або новизною;
- бачити загальну оцінку;
- бачити оцінки основних компонентів;
- бачити головну перевагу та компроміс без відкриття детальної сторінки.

## Межі етапу

Етап 4 не включає Risk Score, ринкову статистику, геокодування, маршрути, AI summaries або дедуплікацію. Ці функції залишаються окремими наступними етапами, щоб Match Score був прозорим і перевірюваним.
